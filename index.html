货代 CRM「客户跟进 + 绩效管理」模块设计

PRDFigma产品设计

+
展开查看完整案例
背景

某货代软件公司面试题。要求在 CRM 中新增跟进记录功能。原题只说「支持销售人员记录跟进活动」。

我的判断

需求末尾提了一句「未来用于绩效评估」。我意识到这个系统有两类用户——销售人员和管理层。如果只设计销售端，管理层的需求根本接不上。于是我主动扩展为双视角设计，同时在数据模型预留 KPI 字段。今天的数据结构，决定明天的迭代空间。

产品方案
  • 销售人员端：跟进录入、多类型事件、历史溯源
  • 销售总监端：绩效看板、勤奋度排行、转化漏斗、异常检测
  • 数据模型预留 KPI 字段，支持未来绩效模块直接接入

海外汽车改装 SaaS · AI 智能客服 Agent

AgentRAGCoze
海外 B2B SaaS · 从业务漏斗流失出发的产品设计
把"Not Supported"断头路变成线索收集器，双引擎架构（Function Calling + RAG）+ MVP 驱动 + 数据飞轮，含安全红线设计
+
展开查看完整案例
为什么要做
  • 沉默的流失：客户查询不支持的车型，遭遇冷冰冰的"Not Supported"页面后直接离开——这是一个每天在发生的漏斗断点
  • 高成本消耗：大量基础实操问题（如故障码查询、刷写步骤）占据了高级技术人员的精力，时差问题让海外用户焦虑等待
破局思路
  • 把"Not Supported"断头路变成线索收集器：Agent 引导用户留下 HW/SW 号，一次失败查询转化为一条研发需求线索
  • Agent 作为 Tier-1 技术支持，用标准化 SOP 处理高频问题，工程师只处理真正复杂的工单
技术架构：双引擎
  • 左脑（确定逻辑）：Function Calling 调用官网 API，100% 准确的车型支持查询，杜绝幻觉
  • 右脑（非结构化知识）：基于 Markdown 的 SOP 知识库（RAG），处理排障指南和情绪安抚
  • 平台选 Coze：低代码快速跑通 MVP，验证 RAG 效果，不需要后端资源投入
核心产品判断

我们没有陷入"大建知识库"的泥潭。首发版本只接入查询 API + 几篇极高频 SOP，跑通主链路比内容完备度更重要——这是 MVP 驱动的产品思维。

数据飞轮设计
  • 前端埋入轻量 👍/👎 反馈，用真实 Badcase 驱动下一篇 SOP 的写作方向
  • 客户缺什么我们补什么，知识库更新变成需求驱动
边界感设计

汽车改装涉及行车安全。System Prompt 明确约束 Agent 只做信息查询和基础 SOP 指导——任何可能涉及硬件损坏的风险，立刻兜底并转人工工单。安全红线不能靠模型自己判断。
